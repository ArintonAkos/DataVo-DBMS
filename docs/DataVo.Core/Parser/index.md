# Parser Overview

The `Parser` module is arguably the most complex component of `DataVo.Core`, responsible for transforming raw, unformatted SQL strings into an executable Abstract Syntax Tree (AST). It acts as the primary compiler for the database, enforcing grammatical correctness, resolving semantic bindings against the catalog, and preparing execution pipelines.

## Core Responsibilities

- **Lexical Tokenization:** The `Lexer` scans raw input strings and converts them into standardized tokens (identifying keywords, literals, and operators).
- **Syntactic Analysis:** The `Parser` builds a hierarchical AST from the tokens, strictly validating that the sequence adheres to the defined SQL grammar.
- **Semantic Binding & Evaluation:** Correlates the parsed identifiers (like column or table names) against the active `Catalog` to ensure they exist and are correctly typed before execution via the `Evaluator`.

## Component Breakdown

| Component (File/Dir) | Architectural Role                                                                                                         |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `AST/`               | Contains the foundational node classes representing abstract expressions.                                                  |
| `Actions/`           | Defines discrete executable steps generated by the parser.                                                                 |
| `Aggregations/`      | Handles the parsing logic for mathematical grouping formulas like `SUM()`, `COUNT()`, or `MAX()`.                          |
| `Binding/`           | Encapsulates the semantic resolution logic validating parsed symbols against the physical catalog.                         |
| `Commands/`          | Defines high-level meta-commands or internal control signals.                                                              |
| `DDL/, DML/, DQL/`   | Specialized parsing subroutines corresponding to Data Definition, Manipulation, and Query languages.                       |
| `Transactions/`      | Executes transactional commands such as `BEGIN`, `COMMIT`, and `ROLLBACK`, including WAL-backed durable commit processing. |
| `Statements/`        | Houses the parsing logic for compound SQL procedural statements.                                                           |
| `Types/`             | Manages the parsing and cast evaluation of raw data scalars into strongly-typed backend equivalents.                       |
| `Utils/`             | Auxiliary parsing utilities and lookahead inspectors.                                                                      |
| `Evaluator.cs`       | Executes the compiled AST against a provided dataset, filtering rows or mapping projections.                               |
| `Lexer.cs`           | The state machine responsible for transforming raw strings into a strict linear sequence of tokens.                        |
| `Parser.cs`          | The central recursive descent parser orchestrating the construction of the entire AST.                                     |
| `QueryEngine.cs`     | The high-level coordinator wrapping the Lexer, Parser, and Evaluator into a single user-facing interface.                  |

## Dependencies & Interactions

This module is the core ingest point for all commands originating from the `Server` module or external clients. It heavily relies on the `Constants` for keyword recognition, the `Enums` for operator switching, and the object hierarchies found in `Models`. During execution it collaborates closely with:

- `Transactions` for explicit transaction state, table-level locking, and durable commit handling.
- `StorageEngine` for physical row persistence.
- `BTree` and `Catalog` for index maintenance and schema validation.
